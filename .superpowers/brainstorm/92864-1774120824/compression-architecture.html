核心架構：壓縮如何操作 ADK Session？

Memory Manager 需要在「ADK 管理對話歷史」的前提下，執行有損壓縮

問題本質

ADK Runner 每輪自動把 Events append 到 Session。但壓縮需要替換舊 Events 為摘要。
這就像 OS 的 page compaction 需要修改 page table——Memory Manager 必須能改寫 Session 的「記憶體」。

A

BeforeModel 攔截（推薦）

Plugin 在 BeforeModelCallback 中改寫送給 LLM 的 request

/* 資料流 */ ADK Session（完整歷史，不動） │ ▼ Runner 準備 LLM Request │ BeforeModelCallback（Memory Manager 攔截） │ ├─ 檢查 memory pressure（S3） ├─ 如果需要壓縮： │ ├─ 從 req.Contents 中取出舊 turns │ ├─ 用獨立 Runner 執行壓縮（S4） │ ├─ 把壓縮結果存到 Plugin 內部 state │ └─ 改寫 req.Contents = [summary] + [recent turns] ├─ 如果不需要壓縮：pass through │ ▼ LLM API Call（只看到壓縮後的 context） │ ▼ AfterModelCallback（讀取 UsageMetadata → S2）

Pros

  • 不修改 ADK Session——完整歷史保留，壓縮只影響送給 LLM 的內容
  • Session 作為 audit log / 完整記錄，壓縮是「view」不是「mutation」
  • 回溯、debug 時可以看到原始對話
  • 實作最簡單——只需操作 LLMRequest

Cons

  • Session 會無限成長（但 MVP 是 single session，可接受）
  • 每次 BeforeModel 都要重新計算「哪些要壓縮」（可 cache）
B

Session 直接修改

壓縮後直接修改 Session 中的 Events——刪除舊 events，插入 summary event

/* 壓縮時 */ Session Events: [e1, e2, e3, e4, e5, ..., e20] │ Compress │ ▼ Session Events: [summary_e1_to_e10, e11, ..., e20] ↑ 舊 events 被替換

Pros

  • Session 大小受控，不會無限成長
  • ADK Runner 自然讀到壓縮後的歷史

Cons

  • ADK SessionService 可能不支援刪除/替換 events
  • 丟失原始對話記錄，無法回溯
  • 需要深入了解 ADK Session 內部實作
C

Custom SessionService + BeforeModel 混合

SessionService 管理壓縮狀態，BeforeModel 做最終組裝

ManagedSessionService ├─ 內部維護：summaries[] + active_events[] ├─ Get() → 返回 [summary_events + active_events] └─ AppendEvent() → 追蹤 + 可能觸發壓縮 Plugin.BeforeModelCallback └─ 確保 req.Contents 使用壓縮後的 view

Pros

  • 最完整的控制
  • 壓縮狀態持久化在 Session 層

Cons

  • 複雜度最高——兩個元件
  • MVP 階段 over-engineering

為什麼推薦 A？

方案 A 的核心洞察：壓縮是一個「view transformation」而非「data mutation」。

這就像資料庫的 materialized view——原始資料完整保留，壓縮只是改變 LLM 看到的「視角」。 這不只是技術簡單性的考量，更是一個架構原則：

  • 原始對話 = source of truth（不可變）
  • 壓縮後 context = derived view（可重算）
  • 任何時候都可以從原始資料重新產生不同的壓縮 view
