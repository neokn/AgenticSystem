專案結構：C4 Level 3 Component View

三種 Go 專案佈局對 Memory Manager 的影響

A

標準 Go Module（可 import 的 Library）

Memory Manager 是獨立 module，其他專案可 go get

agenticsystem/ ├── go.mod # module github.com/neokn/agenticsystem ├── cmd/ │ └── demo/ # demo agent 驗證用 │ └── main.go ├── pkg/ # 公開 API — 其他專案 import 這裡 │ ├── memorymanager/ # 核心：Plugin + MemoryLayout │ │ ├── plugin.go # ADK Plugin 實作 │ │ ├── layout.go # Segment 定義 (S1) │ │ ├── tracker.go # Token tracking (S2) │ │ └── config.go # 配置 │ ├── profile/ # Model Profile Registry (S0) │ ├── compress/ # CompressStrategy interface + 實作 (S3,S4) │ │ ├── strategy.go # interface 定義 │ │ └── generational.go # MVP 策略 │ ├── oom/ # OOM Handler (S6) │ └── metrics/ # Observability └── internal/ # 內部實作細節 └── testutil/

Pros

  • 可被任何 Go + ADK 專案 import
  • 清晰的公開 API 邊界（pkg/）
  • 符合 Go 社群慣例

Cons

  • 需要仔細設計 public API surface
  • MVP 階段 API 可能頻繁變動
B

ADK 應用佈局（internal-first）

以驗證為目的，核心邏輯放 internal/，搭配 demo agent

agenticsystem/ ├── go.mod ├── cmd/ │ └── agent/ # 主程式：ADK Runner + demo agent │ └── main.go ├── internal/ # 全部在 internal — 不承諾公開 API │ ├── memory/ # Memory Manager 核心 │ │ ├── plugin.go │ │ ├── layout.go │ │ └── tracker.go │ ├── profile/ │ ├── compress/ │ │ ├── strategy.go │ │ └── generational.go │ ├── oom/ │ └── metrics/ └── configs/ # Model profiles JSON, 預設配置

Pros

  • API 可以自由變動，不怕 breaking change
  • 聚焦驗證核心邏輯，不用思考 public surface
  • MVP 最快交付

Cons

  • 其他專案無法 import
  • 之後要公開需搬動程式碼
C

混合：internal 開發 → 穩定後 promote 到 pkg

先 internal-first，interface 穩定後再搬到 pkg/ 公開

# Phase 1 (MVP): 全部在 internal/ # Phase 2 (穩定後): 把 interface + config 搬到 pkg/ # pkg/memorymanager/ ← public API (Plugin, Config, Strategy interface) # internal/compress/ ← 策略實作仍為 internal

Pros

  • MVP 速度 = B，未來潛力 = A
  • 自然的演進路徑

Cons

  • 需要紀律——知道何時 promote

📐 Component Dependency（所有方案共用）

profile ← memory (plugin + layout) → compress
memory → metrics
memory → oom
compress → profile (compress worker 需要知道模型)
依賴方向：memory(核心) → profile, compress, metrics, oom
無環狀依賴 ✓
