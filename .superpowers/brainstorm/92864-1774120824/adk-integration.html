Memory Manager × ADK Go 整合方式

ADK 已經有 Plugin 系統和 Session 管理——Memory Manager 該怎麼嵌入？

ADK Go 提供了 AfterModelCallback 可以讀取 UsageMetadata（token 計數）， BeforeModelCallback 可以在 LLM call 前攔截（觸發壓縮）， Session 已經管理對話歷史。

A

ADK Plugin（推薦）

Memory Manager 實作為 plugin.Plugin，掛載到 Runner 上

Runner
 ├─ MemoryManagerPlugin ← 你的 Memory Manager
 │  ├─ AfterModelCallback → 讀 UsageMetadata（S2）
 │  ├─ BeforeModelCallback → 檢查壓力、觸發壓縮（S3）
 │  ├─ OnEventCallback → 追蹤每輪 turn（S2）
 │  └─ BeforeRunCallback → 初始化 Layout（S1）
 ├─ Agent（llmagent）
 └─ SessionService（ADK 內建）

Pros

  • 最乾淨的整合——用 ADK 原生擴展點
  • 自動應用到 Runner 下所有 Agent
  • Plugin callbacks 優先於 agent callbacks

Cons

  • 受限於 Plugin interface 能力
  • Session 的 Event 管理仍由 ADK 控制
B

Custom SessionService Wrapper

包裝 ADK 的 session.Service，在 Session 層面控制歷史

Runner
 ├─ Agent
 └─ ManagedSessionService ← 你的 wrapper
    ├─ Get() → 返回壓縮後的歷史
    ├─ AppendEvent() → 追蹤 + 可能觸發壓縮
    └─ InnerSessionService（原始 ADK service）

Pros

  • 完全控制對話歷史的讀寫
  • 壓縮在 Session 層發生，語意更清楚
  • 可攔截 Get() 返回壓縮後的 events

Cons

  • 需要實作完整的 SessionService interface
  • Token 計數需要另外取得（搭配 Plugin 或 Callback）
C

Plugin + SessionService 混合

Plugin 做感測（token tracking, trigger），SessionService wrapper 做執行（壓縮歷史）

Runner
 ├─ MemoryPlugin（感測層）
 │  ├─ AfterModelCallback → token tracking
 │  └─ BeforeModelCallback → trigger 判斷
 ├─ Agent
 └─ ManagedSessionService（執行層）
    ├─ 持有 MemoryLayout + CompressStrategy
    └─ 被 Plugin 通知後執行壓縮

Pros

  • 關注點分離：感測 vs 執行
  • 各自用最適合的 ADK 擴展點
  • 最大彈性

Cons

  • 兩個元件需要互相通訊
  • 複雜度最高

🔑 關鍵發現

  • ADK 的 resp.UsageMetadata 直接提供 PromptTokenCount / CandidatesTokenCount / TotalTokenCount — 完美對應 S2 的需求
  • ADK Session 已有 Event 概念 — 你的 ConversationTurn 可以映射到 ADK Event
  • Compress Worker 可以用獨立的 llmagent + 獨立 Runner（隔離 context window）
  • ADK 預設支援 Gemini，但標榜 model-agnostic — 需要確認 OpenAI/Anthropic adapter 成熟度
